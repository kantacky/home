変更を安全に小さく分けてコミットするためのスキル。

## 目的

- ステージ状態に依存せず、作業ツリー全体の差分を対象にする。
- コミットを最小粒度に分割する。
- 依存関係を崩さない順序でコミットする。
- コミットメッセージは日本語で、対象差分の要点を過不足なく表現する。

## 手順

### 1. まず「全体差分」を確認する

未ステージ差分だけでなく、ステージ済み差分も必ず確認する。

```bash
git status --short
git diff HEAD
git diff --cached
git diff
```

- `git diff HEAD` を基準に、コミット対象の全体像を掴む。
- `git diff --cached` と `git diff` で、現在のステージ状態による取りこぼしがないか確認する。

### 2. 分割可能性を先に設計する

差分を次の観点で分けられないか検討する。

- 変更の目的が異なるか（例: 振る舞い変更 / リファクタ / テスト追加）
- 変更対象レイヤーが異なるか（例: API / ドメイン / インフラ）
- 単体で意味を持つか（そのコミットだけで理由を説明できるか）

分割候補を作ったら、「各コミットが単体で説明可能か」を再確認する。

### 3. 整合性が保たれる順序を決める

原則として次の順で積む。

1. 土台となる変更（型・インターフェース・スキーマ・共通部品）
2. その土台に依存する実装変更
3. テスト・ドキュメント・補助的修正

- 後続コミットが前コミットに自然に依存する形にする。
- 途中のコミットでビルドやテストが極端に壊れないようにする。

### 4. コミット単位ごとに必要な差分だけをステージする

```bash
git add -p
# または
git add <path>
```

各コミットの直前に、意図した差分だけがステージされているか確認する。

```bash
git diff --cached
git diff
```

### 5. 日本語コミットメッセージを作る

メッセージは必ず日本語で、対象コミットの差分全体を踏まえて書く。

- 何を変えたか
- なぜ必要か
- どこまでをこのコミットで扱うか

例:

```text
APIレスポンス生成処理を共通化し、重複ロジックを削減

レスポンス組み立て処理が各ハンドラに分散していたため、
共通関数へ集約して重複実装を削除した。
このコミットでは振る舞いは変えず、既存テストが通る範囲の
リファクタリングのみを行っている。
```

### 6. コミット後に残差分を確認して次へ進む

```bash
git show --stat --oneline -1
git status --short
git diff HEAD
```

- 直前コミットの内容が意図通りか確認する。
- 未コミット差分が残っていれば、同じ手順で次の最小単位に分けて進める。

## 禁止事項

- ステージ済み差分だけ見てコミット判断すること。
- 複数目的の変更を「ついで」で同一コミットに混ぜること。
- 英語コミットメッセージで記録すること。
- 差分の一部しか説明していない曖昧なメッセージを書くこと。
